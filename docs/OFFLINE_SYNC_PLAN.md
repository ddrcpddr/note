# 离线记录与恢复同步方案

更新时间：2026-07-04

当前项目是家庭自用工具，离线能力的第一目标不是复杂协同，而是避免手机临时记录因为 NAS / Docker 服务不可用而丢失。

## 第一版范围

- 支持新建记录在服务不可用时保存到浏览器本地。
- 本地记录显示“待同步到 NAS”。
- 服务恢复后自动把本地队列逐条写入 /api/notes。
- 同步成功后用服务端正式记录替换本地临时记录。
- 富文本 HTML / JSON、纯文本、分类、成员、标签、图片和附件草稿都会进入待同步 payload。

## 暂不做的范围

- 离线编辑已存在的服务端记录。
- 离线删除 / 归档记录。
- 多设备冲突合并。
- 后台 Service Worker 自动同步。
- Android 原生离线数据库。

这些能力等 Web 端新建队列稳定后再做。

## 本地存储策略

- 当前使用 localStorage 保存 home-notes-offline-create-queue-v1。
- 队列项包含 localId、payload、用于页面展示的 note、createdAt。
- payload 与 /api/notes 创建接口保持一致，方便恢复后直接 POST。
- 如果浏览器存储配额不足，页面不崩溃，但该条记录可能无法持久化，需要后续 UI 提示增强。

## 同步规则

1. App 启动时读取本地队列。
2. 如果服务不可用，首页和详情仍显示本地待同步记录。
3. 如果服务恢复为 sqlite 模式，自动执行同步。
4. 同步按队列顺序逐条 POST 到 /api/notes。
5. 每成功一条，就从本地队列移除一条，并替换对应本地临时记录。
6. 如果某条失败，停止后续同步，保留剩余队列，避免丢失。

## 冲突策略

- 第一版只处理“本地新建记录”，没有服务端同一记录的并发修改。
- 本地临时记录未同步前可以继续停留在页面，但暂不保证离线编辑后覆盖原队列项。
- 后续如果做离线编辑，需要给本地记录加版本号和 dirty 字段。

## 人工测试建议

1. 关闭 Docker 或断开服务。
2. 打开前端，新建一条富文本记录，插入图片或附件。
3. 确认详情页显示“待同步到 NAS”。
4. 刷新页面，确认本地记录仍存在。
5. 恢复 Docker 服务。
6. 刷新或等待页面恢复，确认记录自动同步并变成正式记录。

## 后续增强

- 改用 IndexedDB 存储大附件，避免 localStorage 容量限制。
- 增加“手动同步”按钮和失败原因提示。
- 支持离线编辑本地待同步记录。
- 支持服务端记录离线编辑时的冲突检测。

## PWA 前端壳缓存补充

当前已新增 service worker，用于缓存前端壳和静态资源。它不缓存 /api/ 数据。

这意味着：

- 手机或浏览器至少在线打开过一次家事记后，后续 Docker/NAS 暂时不可用时仍有机会打开已缓存的前端壳。
- 离线时新建记录进入本地待同步队列。
- 服务恢复后，前端再把队列同步到 /api/notes。
- 如果从未在线打开过页面，或者浏览器清理了站点缓存，Docker/NAS 不在线时仍无法冷启动页面。
