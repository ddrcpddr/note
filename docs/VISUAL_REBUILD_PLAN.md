# Visual Rebuild Plan

## 1. 结论

当前停止继续做“在现有 React / Tailwind 代码上微调 CSS”的视觉对齐方式。

这条路线已经失败：它只能局部修掉某个溢出、字号或遮挡问题，但无法稳定还原 `design/home-records-prototype/` 中 7 张 Product Design 最终图的页面比例、组件层级、留白节奏、底部导航位置和卡片内部结构。后续如果继续凭感觉改 class，只会反复出现分类页文字挤压、首页底部遮挡、设置页技术感过重等问题。

后续必须改为：按 Product Design 7 张最终图重建页面视觉层。

执行前提：没有 Figma Agent 输出的精确页面规格，不继续写 UI 代码；没有确认素材来源，不自己画临时 SVG 或继续拼凑丑图。

本轮只创建方案文档，不修改业务代码、不修改数据库、不修改真实 Note Station 数据、不生成图片、不调用 Product Design。

## 2. 为什么当前微调方式失败

### 2.1 缺少可执行的尺寸基准

当前代码主要依赖 Tailwind class 直接堆在页面 JSX 里，例如 `px-*`、`gap-*`、`text-[...]`、`min-h-[...]` 分散在各页面。每次调整只能凭肉眼和局部截图判断，没有一份来自 Figma 的精确规格表约束：

- 页面左右边距；
- 顶部标题区高度；
- 搜索框高度和位置；
- 分类卡片宽高；
- 图标圆形底尺寸；
- 字号、行高、字重；
- 底部导航和浮动按钮避让区；
- 长标题、长分类名、长时间文本的处理规则。

结果是一个页面局部修好，另一个元素又被挤压或比例失衡。

### 2.2 现有组件不是按最终图组件体系拆的

当前页面实现混合了真实功能状态、早期 V1 原型、Figma handoff 文档和后续 bugfix 结果。它能跑，但视觉层不是按最终 7 张图的组件边界重建的。

例如分类页卡片曾在两种错误方向之间来回：

- 横向卡片时文字区被图标、gap、箭头挤窄，出现省略号或竖排风险；
- 改成上图标下文字后虽然解决挤压，但偏离 Product Design 最终图的横向卡片结构。

这说明问题不是单个 CSS 值，而是缺少“最终图组件规格 -> 代码组件”的映射。

### 2.3 当前截图验收没有形成硬门槛

之前的检查更多是“无横向溢出”和“脚本不报错”，但这不足以证明还原最终图。后续验收必须以截图对比为准：

- 当前页面截图必须和对应最终图并排检查；
- 390px 和 430px 都要检查；
- 只要出现标题省略号、文字竖排、主要按钮遮挡、底部导航压内容、卡片比例明显失衡，就不能进入下一页。

| P0 | 设置 / NAS 存储与备份 | 是 | 技术信息密度高，生活化装饰和卡片层级与最终图差距大。 |

## 6. 主 Codex 后续如何逐页重建

### 6.1 总原则

- 不再全站统一调 CSS。
- 不改业务逻辑、不改数据库、不改真实 Note Station 数据。
- 每次只重建一个页面的视觉层。
- 页面功能仍以当前代码为准，视觉结构以 Product Design 最终图为准。
- 每页单独 commit，失败不进入下一页。
- 每页完成后必须截图对比，不以“看起来接近”作为通过依据。

### 6.2 推荐实现方式

1. 建立页面级视觉组件
   - 为每个主要页面拆出独立视觉区块，例如 `CategoryPageVisual`、`HomeTimelineVisual`。
   - 保留现有数据 props 和事件 handler，不改 API。
   - 先在页面内部重建结构，避免动全局样式造成其他页面漂移。

2. 建立最终图 token
   - 根据 Figma Agent 输出创建局部 token：spacing、radius、font、shadow、nav/fab offset。
   - token 只能服务 7 图还原，不引入新的视觉方向。

3. 页面逐个替换
   - 从分类页开始；
   - 完成一个页面后运行 check/test/build；
   - Docker 3300 重建；
   - Playwright 截图；
   - 对照最终图人工确认；
   - 更新审计文档；
   - commit。

4. 不做跨页顺手重构
   - 分类页修复不顺手改首页；
   - 首页修复不顺手改搜索；
   - 设置页修复不顺手改导入。

## 7. 每页验收标准

### 7.1 分类页

基准图：`design/home-records-prototype/page-5-categories.png`

必须通过：

- 11 个分类完整显示：家庭事务、房屋 / 设备、维修 / 售后、购物 / 消费、证件 / 账号、孩子 / 教育、老人 / 健康、宠物、工作 / 杂事、临时记录、未分类；

- 底部导航不遮挡最后一行；
- 浮动按钮不遮挡分类卡片主要内容。

### 7.2 首页 / 生活时间线

基准图：`design/home-records-prototype/page-1-home-selected.png`

## 8. 每页完成后如何截图对比

每页必须产出以下本地临时截图，不提交 Git：

```text
C:\tmp\note-visual-rebuild\<page>-390.png
C:\tmp\note-visual-rebuild\<page>-430.png
```

每页检查步骤：

1. 运行本地生产服务或 Docker 3300；
2. Playwright 以 390px 和 430px 视口打开对应页面；
3. 截图保存到 `C:\tmp\note-visual-rebuild\`；
4. 与 `design/home-records-prototype/page-*.png` 并排人工检查；
5. 脚本检查：
   - `document.documentElement.scrollWidth <= window.innerWidth`；
   - 页面关键标题完整；
   - 没有 `...` 形式的关键分类 / 标题省略；
   - 没有关键文本多行竖排；
   - 底部导航和浮动按钮不遮挡主要内容；
6. 将结论写入 `docs/PRODUCT_DESIGN_RESTORE_AUDIT.md`。

如果任一页面失败，不进入下一页。

## 9. 后续执行门槛

后续开始写 UI 代码前必须满足：

1. 使用已存在的 `docs/FIGMA_IMPLEMENTATION_SPECS.md` 作为页面尺寸、组件尺寸、字体、间距和锚点基准；如某页仍缺细节，先让 Figma Agent 补充；
2. 确认 `design/image-assets/v1/` 中现有素材是否足够，尤其是设置页生活装饰是否需要补图；
3. 用户确认从哪个页面开始重建，建议顺序为：分类页 -> 首页 -> 设置页 -> 搜索页 -> 新建页 -> 详情页 -> 导入页；
4. 明确每页只改视觉层，不改业务逻辑；
5. 每页独立截图验收、独立 commit。

在以上门槛满足前，主 Codex 停止继续凭感觉调整 UI。
